Неотъемлемыми свойствами ПО являются : 
незримость, согласованность, изменяемость, сложность

В основе программной инженерии лежит одна фундаментальная идея: проектирование ПО является 
формальным процессом, который можно изучать и совершенствовать

большим количеством разных элементов 

Изменяемость ПО состоит в том, что
его необходимо адаптировать к изменениям технических средств и программного окружения; возникает необходимость у пользователей в расширении его функционльности 

По оценке Коберна, быстрая разработка ПО применима только в проектах 
среднего масштаба, критичность D; малого масштаба, критичность C; малого масштаба, критичность D; среднего масштаба, критичность С  

Под проектом ПО будем понимать совокупность 
спецификаций ПО (включающих модели и проектную документацию), обеспечивающих создание ПО в конкретной программно-технической среде 

Согласованность ПО состоит в том, что 
оно должно согласовываться с уже существующим ПО

Программное обеспечение определяется как набор : 
данных, документации, процедур, компьютерных программ

//Тест 8. Анализ и проектирование ПО
Объектно-ориентированное проектирование включает два вида деятельности:
проектирование элементов системы; проектирование архитектуры системы

Проектирование архитектуры системы выполняется архитектором системы и включает в себя:
идентификацию архитектурных решений и механизмов, необходимых для проектирования системы; анализ взаимодействий между классами анализа, выявление подсистем и интерфейсов; формирование архитектурных уровней; проектирование конфигурации системы; проектирование структуры потоков управления

Проектирование классов включает следующие действия:
уточнение операций и атрибутов; детализация проектных классов; уточнение связей между классами; моделирование состояний для объектов классов

Анализ вариантов использования выполняется проектировщиками и включает в себя:
определение атрибутов и ассоциаций классов; идентификацию классов, участвующих в реализации потоков событий варианта использования; распределение поведения, реализуемого вариантом использования, между классами (определение обязанностей классов); унификацию классов анализа

Архитектурные механизмы отражают
нефункциональные требования к системе; реализацию функциональных требований в архитектуре системы

Граничные классы (Boundary) —
служат посредниками при взаимодействии внешних объектов с системой

Объектно-ориентированный анализ включает два вида деятельности: и
архитектурный анализ; анализ вариантов использования

Укажите обязанности для каждого типа классов:
классы-сущности отвечают за хранение и манипулирование данными; граничные классы отвечают за взаимодействие с внешней средой системы; управляющие классы координируют потоки событий варианта использования

Целью объектно-ориентированного анализа является
создание стабильной основы архитектуры системы; трансформация функциональных требований к ПО в предварительный системный проект	

Проектирование структуры потоков управления выполняется
при наличии в системе параллельных процессов (параллелизма)

В потоках событий варианта использования выявляются классы трех типов
Классы-сущности (Entity); Граничные классы (Boundary); Управляющие классы (Control)

Классы-сущности (Entity) —
представляют собой основные абстракции (понятия) разрабатываемой системы, рассматриваемые в рамках конкретного варианта использования

Установите соответствие 8
механизмы — это некоторые типовые решения (образцы), они документируются в проекте (модели) с помощью кооперации; поведение описывается с помощью диаграмм классов; структурная часть механизма описывается с помощью диаграмм взаимодействия

Проектирование элементов системы выполняется проекти­ровщиками и включает в себя:
проектирование баз данных; уточнение описания вариантов использования; проектирование классов

Соглашения моделирования определяют:
организацию модели (пакеты); используемые диаграммы и элементы модели; правила применения элементов модели; соглашения по именованию элементов модели

Управляющие классы (Control) —
обеспечивают координацию поведения объектов в системе

Первым действием архитектора при выявлении подсистем является преобразование классов анализа в проектные классы (design classes). По каждому классу анализа принимается одно из двух решений:
сложный класс анализа может быть разбит на несколько классов, преобразован в пакет или в подсистему; класс анализа отображается в проектный класс, если он простой или представляет единственную логическую абстракцию

Архитектурный анализ выполняется архитектором системы и включает в себя:
формирование начального представления архитектурных уровней; утверждение общих стандартов (соглашений) моделирования и документирования системы; предварительное выявление архитектурных механизмов (механизмов анализа); формирование набора основных абстракций предметной области (классов анализа)

Архитектура крупномасштабной системы должна удовлетворять следующим требованиям:
размер компонентов не должен быть слишком большим; компоненты системы должны иметь возможность замены; однородные функции должны группироваться вместе; e. изменения в одних компонентах не должны сильно затрагивать другие
